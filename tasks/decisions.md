# 技术决策记录

## TDR-001：AI 服务商选择 DeepSeek
- **日期**：2026-03-17
- **决策**：使用 DeepSeek API v3 作为主力 AI 服务
- **理由**：国内调用稳定无需代理、性价比极高（约 GPT-4 1/10 价格）、中文公考内容理解强
- **影响**：所有 AI 调用封装在 `DeepSeekClient`，接口设计为抽象层，未来可切换其他模型

## TDR-002：OCR 方案采用 Python 微服务
- **日期**：2026-03-17
- **决策**：PaddleOCR 独立为 Flask 微服务（端口 5001），SpringBoot 通过 HTTP 调用
- **理由**：Java 调用 Python C++ 扩展太复杂；微服务解耦，可独立扩缩容
- **影响**：Docker Compose 需包含 ocr-service 容器

## TDR-003：文件存储使用 MinIO
- **日期**：2026-03-17
- **决策**：用户头像和笔记图片存储到 MinIO
- **理由**：本地部署兼容 S3 协议，开发/生产环境一致；避免直接存文件系统导致部署困难
- **影响**：需要 MinIO Docker 容器，SpringBoot 引入 minio SDK

## TDR-005：SRS 算法采用简化 SM-2
- **日期**：2026-03-17
- **决策**：间隔复习算法：会了 → interval = min(ceil(interval × 2.5), 60)；不会 → interval = 1
- **理由**：完整 SM-2 需维护 E-Factor 参数，实现复杂；简化版对学习效果影响不大，适合本项目规模
- **影响**：SrsService 被错题本和抽认卡共同依赖，必须作为无状态工具类实现，不能持有实体状态

## TDR-006：模考超时以服务端检测为准
- **日期**：2026-03-17
- **决策**：服务端在 GET /api/mock/{examId} 时检测 started_at + time_limit 是否超时，前端倒计时仅用于展示
- **理由**：纯前端计时可被用户修改系统时间绕过；服务端兜底更可靠
- **影响**：前端需每 30 秒轮询同步剩余时间；Spring Scheduled 任务每 5 分钟扫描超时模考

## TDR-007：站内通知用 Redis List 而非数据库表
- **日期**：2026-03-17
- **决策**：管理员处理反馈后的通知写入 Redis List（key: notice:{userId}，TTL 7天），不建数据库表
- **理由**：通知量少、时效短，Redis 足够支撑；减少数据库表数量，降低维护成本
- **影响**：通知不会持久化超过 7 天；未来若需消息中心再迁移数据库
